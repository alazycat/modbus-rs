Issue #0033 — Sync UDP client for coils & registers — 2026-07-07

Design Decisions

Decision Validate UDP response source address

Ambiguity: UDP is connectionless, so a received datagram could come from any host.

Choice: UdpTransport::recv captures the peer address from recv_from and returns TransportError::Disconnected if it does not match the configured remote.

Rationale: Prevents accepting spoofed or stray responses. The check is done at the transport layer so the client can rely on transaction/unit ID validation alone.

Decision Per-call recv timeout via socket read timeout

Ambiguity: The Transport::recvUdpSocket only supports a global read timeout.

Choice: Set the socket read timeout to the supplied duration on each recv call; treat Duration::ZERO as an immediate timeout without touching the socket.

Rationale: Fits the existing Transport contract and avoids platform-specific rejection of zero-length timeouts. It does overwrite any timeout the caller configured on the socket; this is documented implicitly by the trait.

Deviations

Deviation Aligned TCP client error formatting

Spec said: Issue #33 is about UDP client only.

Implemented: Also updated TcpClientError to use Display formatting and chain EncodeError/DecodeError through Error::source, matching the new UDP client.

Why: The two error enums were otherwise identical; keeping them consistent reduces future maintenance burden.

Tradeoffs

Tradeoff Fixed-size receive buffer

Alternatives: Allocate a Vec sized from the UDP datagram length, or peek the datagram size first.

Chosen: A 512-byte stack buffer in the client and transport, matching the TCP/RTU/ASCII clients.

Why: Keeps the implementation uniform and allocation-free on the hot path. The downside is that a max-length Modbus response may not fit; current PDU lengths stay well below this limit.
