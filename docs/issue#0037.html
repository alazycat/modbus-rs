Issue #37 — Async TCP client/server for coils & registers — 2026-07-07

Design Decisions

Decision Colocated async TCP code with sync TCP code

Ambiguity: The spec did not specify module layout for the new async TCP types.

Choice: Added AsyncTcpTransport, AsyncTcpClient, and AsyncTcpServer to the existing tcp_transport.rs, tcp_client.rs, and tcp_server.rs modules using #[cfg(feature = "async")] gates.

Rationale: This mirrors the established UDP and ASCII module structure, keeps related transport/client/server logic together, and avoids proliferating files.

Decision Generic async transport over AsyncReadExt + AsyncWriteExt

Ambiguity: The acceptance criteria say "Async TCP transport" without defining whether it must be bound to TcpStream.

Choice: Implemented AsyncTcpTransport<T> for any T: AsyncReadExt + AsyncWriteExt + Unpin + Send.

Rationale: A generic stream interface is testable with tokio::io::duplex and reusable for TLS wrappers or other stream-like types, matching the flexibility of the sync TcpTransport.

Deviations

None — implementation followed the issue acceptance criteria and existing codebase patterns.

Tradeoffs

Tradeoff Same-file colocation vs. separate async modules

Alternative A: Create dedicated async_tcp_*.rs files. Pros: cleaner separation of sync and async code; easier to gate entire files. Cons: diverges from UDP/ASCII precedent; more files to navigate.

Alternative B: Colocate async code in existing files. Pros: consistent with UDP/ASCII; keeps TCP-specific logic in one place. Cons: files are longer; mixed cfg gates.

Chosen: Alternative B for consistency with the codebase.
