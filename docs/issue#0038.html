Implementation Notes

Issue #0038 — Async RTU client/server for coils & registers — 2026-07-07

Design Decisions

Decision Reused the generic async client/server instead of adding RTU-specific types

Ambiguity: The issue calls for an "async RTU client/server," but the existing async runtime already defines protocol-agnostic AsyncClient<T> and AsyncServer<D>.

Choice: Added AsyncRtuTransport<T> and the alias AsyncRtuClient<T>; the server side continues to use AsyncServer<D> directly.

Rationale: RTU framing lives in the transport layer. Duplicating client/server logic would add no behavior and would diverge from the TCP/UDP/ASCII pattern. The alias gives users a discoverable name for the client half without inventing a misleading server wrapper.

Decision Async recv uses an overall frame deadline

Ambiguity: RTU framing is traditionally delimited by inter-frame silence, which could suggest resetting the timeout after every byte.

Choice: The async transport computes one deadline from timeout (tokio::time::timeout_at) and does not refresh it per byte.

Rationale: AsyncTransport::recv is documented to return a complete frame within the supplied timeout. This matches the async TCP and ASCII transports and gives callers a predictable upper bound. Users who need inter-character silence detection can configure that on the underlying stream.

Decision Added a 256-byte cap on RTU frame accumulation

Ambiguity: The byte-by-byte framing loop could grow its buffer without bound on a stream that never decodes.

Choice: Added RtuAdu::MAX_FRAME_SIZE = 256 and enforced it in sync and async RtuTransport, RtuServer, and AsyncServer.

Rationale: A standard Modbus RTU ADU cannot exceed 256 bytes. Capping memory protects against broken or malicious peers and keeps the four RTU frame readers consistent.

Decision Made open_serial_rtu an async function

Ambiguity: The helper currently contains no .await points.

Choice: Kept it async for symmetry with the other transport helpers and to leave room for async serial-port setup in the future without a breaking change.

Rationale: An async API signals that it is meant for the async runtime, even if the current implementation is synchronous under the hood.

Deviations

Deviation No dedicated AsyncRtuServer type

Spec said: "Async RTU client/server" could be read as requiring paired client and server types.

Implemented instead: Only AsyncRtuClient<T> is exposed; servers use the generic AsyncServer<D>. An initial AsyncRtuServer<D> alias was added and then removed because it bound to no RTU-specific behavior and was inconsistent with TCP/UDP/ASCII.

Why: The generic async server already accepts any AsyncTransport via serve_one/serve. A type alias that did not mention AsyncRtuTransport was more confusing than helpful.

Deviation Serial helper uses fixed Modbus defaults

Spec said: "tokio-serial wrapper."

Implemented instead: A single convenience function open_serial_rtu(path, baud_rate) that hard-codes 8 data bits, no parity, 1 stop bit, and a 100 ms timeout.

Why: Those defaults cover the common RTU serial configuration. Users needing custom settings can open a SerialStream themselves and wrap it with AsyncRtuTransport::new.

Tradeoffs

Tradeoff Byte-by-byte reads vs. vectored reads

Alternatives: Read large chunks and scan for frame boundaries, or read one byte at a time.

Pros/cons: Chunked reads are faster but require careful buffer management and do not simplify RTU framing much because the valid frame length is unknown until decoded. Byte-by-byte reads are simpler and mirror the sync RtuTransport.

Why this won: Modbus RTU frames are small; simplicity and symmetry with the existing sync transport outweighed the modest throughput gain of chunked reads.

Tradeoff Overall timeout vs. per-byte timeout

Alternatives: Reset timeout after each received byte (inter-character timeout) or use it as a total frame budget.

Pros/cons: Per-byte reset matches classic RTU silence timers but makes the worst-case receive time unbounded (N × timeout). An overall deadline gives callers a hard bound but can time out a slow valid frame.

Why this won: The AsyncTransport contract specifies a frame-level timeout, and the async TCP/ASCII transports already use that semantics. A hard bound is less surprising for library users.

Open Questions

Question Should AsyncRtuServer be reintroduced as a newtype?

We removed the alias because it was not type-safe. If discoverability is important, a thin AsyncRtuServer<D>(AsyncServer<D>) newtype with only serve_one/serve forwarding could be added later.

Question Should open_serial_rtu accept more configuration?

Currently parity, stop bits, and data bits are hard-coded. If users commonly need other settings, consider a builder or accepting a preconfigured tokio_serial::SerialPortBuilder.

Question Does the overall-frame timeout work for real serial ports?

The timeout bounds the whole recv call. For serial ports that deliver bytes slowly, this may be tighter than an inter-character timeout. Verify against target hardware or add a separate inter-character timeout parameter if needed.
