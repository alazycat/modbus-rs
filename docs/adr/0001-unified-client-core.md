# ADR-0001: Sync/async client cores share method generation via macros

## Status

Accepted

## Context

The `modbus` crate exposes both synchronous and asynchronous Modbus clients. The current implementation maintains two nearly identical client cores:

- `ClientCore<A: AduAdapter>` in `src/client/sync.rs`
- `AsyncClientCore<A: AsyncAduAdapter>` in `src/client/async.rs`

Both cores contain the same `dispatch` logic, the same function-code methods (generated by `impl_client_methods!`), and the same typed register helper methods (`read_holding_registers_u32`, `write_multiple_registers_string`, etc.). Any change to dispatch, a new function code, or a typed helper has to be edited in two places. This fractures locality and makes the interface of each core nearly as wide as the implementation behind it.

The crate also supports:

- `no_std` + `alloc` builds
- independent `sync` and `async` feature flags
- not pulling `tokio` into sync-only builds

These constraints rule out designs that make the synchronous client depend on an asynchronous runtime.

## Decision

We will keep sync and async clients as **compile-time variants** generated from shared macros, rather than unifying them into a single async-first core or a single generic core.

Specifically:

1. `ClientCore` and `AsyncClientCore` remain two distinct types because their adapter seams differ (`AduAdapter` vs `AsyncAduAdapter`).
2. All duplicated method bodies — `dispatch`, function-code methods, and typed helpers — are generated from macros in `src/client/macros.rs`.
3. `impl_client_methods!` moves from `src/macros.rs` to `src/client/macros.rs` because it is client-specific.
4. Core client behavior tests move to `src/client/mod.rs` and exercise both sync and async through a shared mock adapter at the `AduAdapter` seam.
5. Transport-specific connection helpers (e.g. `connect_rtu_over_tcp`) remain hand-written per runtime, because they depend on concrete sync/async I/O APIs.
6. The refactor stays scoped to the client; the server already centralizes dispatch in `Server::dispatch` and is not affected.

## Consequences

- **Locality improves**: client bugs, new function codes, and typed helper changes concentrate in one macro definition or one macro invocation.
- **Leverage improves**: the same macro call generates behavior for both runtimes.
- **Tests improve**: core behavior is verified once through the adapter seam instead of being duplicated.
- `src/client/sync.rs` and `src/client/async.rs` become thin shells: struct declaration, constructors, macro calls, and runtime-specific connection helpers.
- The public API remains unchanged; callers still use `Client::new(...)` and `AsyncClient::new(...)`.

## Rejected alternatives

- **Async-first core with blocking adapter**: would have unified the code most completely, but would force `tokio` into sync-only builds and break the current feature contract.
- **Single generic `ClientCore<A, Runtime>`**: would have abstracted over sync/async with a runtime marker type, but added a new concept without reducing duplication enough to justify it. The `AduAdapter` / `AsyncAduAdapter` seam already expresses the runtime distinction.
- **Expanding the refactor to the server**: the server duplication is small because `Server::dispatch` is already a single deep module used by `AsyncServer`. Widening scope would add risk for little gain.
