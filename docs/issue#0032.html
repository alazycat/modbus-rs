Implementation Notes

Issue #0032 — Sync ASCII server for coils & registers — 2026-07-07

Design Decisions

Decision Propagate malformed ASCII frames as Decode errors

Ambiguity: The issue did not specify whether a malformed ASCII frame should be treated as a peer disconnect or as a decoding failure.

Choice: Added an AsciiServerError::Decode variant and return it for frames that are complete (end with \r\n) but fail LRC or length checks. EOF mid-frame is still reported as Disconnected.

Rationale: This matches the synchronous TCP server, which also distinguishes decode errors from disconnects, making debugging easier and avoiding silent swallowing of corrupt data.

Decision Shared ASCII framing constants

Ambiguity: The ASCII framer did not define broadcast semantics or a minimum frame size, while the RTU framer did.

Choice: Added AsciiAdu::MIN_FRAME_SIZE, AsciiAdu::BROADCAST_ADDRESS, and AsciiAdu::is_broadcast() to src/ascii.rs and used them in the server.

Rationale: Keeps the ASCII and RTU APIs consistent and removes magic numbers from the server.

Deviations

None — implementation followed the acceptance criteria as written.

Tradeoffs

Tradeoff Fixed 513-byte transmit buffer

Alternatives: A dynamically sized Vec sized from the encoded length, or a shared constant derived from the max PDU size.

Chosen: A stack buffer of 513 bytes, which is exactly enough for a 253-byte PDU wrapped in an ASCII ADU.

Why: It keeps the server simple and allocation-free on the response path. The downside is that a future increase in the max PDU length must also update this buffer; a named constant or dynamic buffer would be safer if the protocol limits change.

Tradeoff Reused lrc8 helper in AsciiAdu::encode

Alternatives: Keep the existing inline sum, or add a zero-allocation slice helper.

Chosen: Build a small temporary Vec containing address + PDU and call lrc8.

Why: It removes duplicate LRC logic and makes the implementation easier to verify. The cost is a small per-encode allocation; for the current use cases this is acceptable.

Open Questions

Question Maximum response PDU enforcement

The server currently trusts that Server::dispatch will not produce a PDU larger than the buffer can encode. Should the server explicitly guard or truncate oversized responses?

Question Serial half-duplex turnaround

ASCII is often run over RS-485 half-duplex links that require a transmit-enable turnaround delay. Should the server expose a hook for post-send turnaround, or is that the transport layer's responsibility?
